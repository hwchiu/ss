SHARED LANGUAGE

Serverless 不是「沒有伺服器」

它是一組平台承諾：抽象基礎設施、依需求伸縮、事件驅動、按實際使用承擔成本。

開發者交付程式碼或容器不管理 node、pod 與 ingress
平台負責Build、部署、路由、伸縮以 policy 與 identity 統一治理
工作負載取得穩定 endpoint 或事件來源流量與 revision 分離
01

Request-driven

HTTP/gRPC 請求驅動服務實例。沒有流量時可縮到零，但入口必須能等待新實例 ready。

典型：API、網站、Webhook
02

Event-driven

Queue、stream 或 cloud event 驅動 consumer。伸縮依據是 backlog，而不是單看 CPU。

典型：Kafka、SQS、RabbitMQ
03

Run-to-completion

一次執行完成後退出，支援平行度、逾時、重試與執行歷史，不應偽裝成常駐服務。

典型：批次、排程、資料轉換
04

Always-on worker

主動拉取事件或維持連線，不一定適合 scale-to-zero，需要獨立的生命週期與計價思考。

典型：Kafka consumer、stream

一次部署背後發生什麼

Serverless 的價值來自整條供應鏈，不只是 autoscaling。

常見誤解

裝了 KEDA 就有 Serverless

KEDA 解決事件指標與 HPA 伸縮；沒有處理 revision、入口、build、runtime、traffic 與開發者介面。

常見誤解

Scale-to-zero 就一定省錢

冷啟動、預留容量、broker、gateway 與控制面仍有成本；需以完整 workload economics 評估。

常見誤解

Namespace 等於多租戶隔離

還需要 quota、network policy、workload identity、admission policy、runtime sandbox 與獨立 node pool 策略。

關鍵設計原則

  1. 01

    穩定 endpoint，不可變 revision部署與流量操作彼此解耦。

  2. 02

    控制面失效，資料面繼續服務避免平台 API 成為線上請求依賴。

  3. 03

    事件採 at-least-once 設計提供 retry、DLQ、replay 與冪等鍵。
